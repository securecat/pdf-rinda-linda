# 設計メモ：マルチステップOCRパイプライン

作成：2026-06-30

---

## 背景・問題意識

### 現在のアプローチの限界

現状は「ページ画像1枚をClaudeに渡して、1回のプロンプトでMarkdown化」という方式。これは以下の理由で限界がある。

**「絵として成立する文字」問題**

大きくデザインされたテキスト（例：ミッションステートメント "誰一人取り残されない、人に優しいデジタル化を。"）は、視覚的には「ビジュアルデザインの一部」に見える。Claude が：

- テキストとして転写する → ミッション文を正しく読める
- 絵の一部として無視する → 現状の問題（テキストが丸ごと消える）

どちらが正しいかはコンテンツによって異なる。ロゴ・タイポグラフィアートは「絵」のままが正しいし、大きく組まれたミッション文は「テキスト」として読むべき。これを1回のプロンプトで判別させるのは構造的に無理がある。

### 具体的な未解決ケース（デジタル庁BRIDGEパンフ p.02）

- Mission の巨大テキスト → 完全欠落
- Vision の2段組本文 → ほぼ欠落
- Values の4段組本文 → 完全欠落

プロンプト改善（---→—、省略禁止指示など）を重ねても改善しなかった。根本がアーキテクチャの問題のため。

## 提案するアーキテクチャ

### 全体の流れ

```
Step 1: クラスター認識（全体画像）
         ↓
Step 2: クラスターごとに分類・処理（要素単位）
         ↓
Step 3: Markdown再構成（全体画像＋全クラスター結果）
```

### Step 1：クラスター認識

- **入力**：ページ全体画像（見開きの場合は分割前の全体）
- **処理**：ページ内の「要素のまとまり」を列挙する
- **出力**：「このページには〇〇なまとまりが△個ある」という構造的記述

ここでは内容の転写はしない。「何がどこにある」の把握のみ。

### Step 2：クラスターごとの分類・処理

各クラスターに対して：

1. **分類**：「テキストとして転写すべき」か「画像として説明すべき」かを判断
2. **処理**：
   - テキスト → OCR・転写
   - 画像 → `[画像: ...]` 記述

分類の判断基準（案）：
- 文章として意味をなすか（句読点・助詞があればテキスト）
- レイアウト上のテキストエリアに属しているか
- 読まれることを前提とした配置か（本のような左右の流れ）
- vs. ロゴ・アイコン・アート的な配置

### Step 3：Markdown 再構成

- **入力**：全体画像 ＋ Step 2 の全クラスター処理結果
- **処理**：見出し階層・文書構造を全体レイアウトから推測してMarkdown化
- **理由**：`#` / `##` / `###` の階層は全体を見ないと正しく決まらないため、最後にまとめて行う

---

## 見開きページとの関係

| ステップ | 使用画像 |
|---|---|
| Step 1（クラスター認識） | 全体（分割前）|
| Step 2（クラスターごと処理） | 左右分割した各画像（クラスターが属する側） |
| Step 3（Markdown再構成） | 全体（または左右それぞれ） |

## 実装上の課題・未解決問題

### クラスターを「切り出せない」問題

Claude はピクセル座標を返せないため、Step 2 での「クラスターごと処理」を実際に画像トリミングして送ることができない。

代替案：
- **全体画像 ＋ 言語的フォーカス指示**：「左上のMissionセクションに注目して…」
- Step 1 の出力を言語的な領域記述として使い、Step 2 でそれを参照させる

→ これで精度が出るかどうかが最大の不確実性。

### API コストの増加

| 方式 | 1見開きあたりのAPIコール数 |
|---|---|
| 現在 | 2回（左右各1回） |
| 提案 | 4〜6回程度（Step1×1 ＋ Step2×N ＋ Step3×1） |

ページ数が多いドキュメントではコストが3倍程度になる可能性がある。

### Step 1の粒度設計

クラスターをどの粒度で認識するかが難しい：
- 粗すぎる → Step 2 で結局同じ問題が出る
- 細かすぎる → API コールが増えすぎる

---

## v1.0.0 における位置づけ

このパイプラインの実現が **v1.0.0 の必要条件**。

v1.0.0 の定義：「1ページを、どんなレイアウト・内容でも、誰もが平等に読めるようにする」。  
現行の単一プロンプト方式では"どんなレイアウトでも"を達成できないことが実験で確認されたため、このアーキテクチャへの移行が前提となる。

多ページ対応（将来構想）は v1.0.0 の後。

## 活用できる可能性があるライブラリ・技術（未検討）

Chrome拡張という制約（Node.jsネイティブモジュール不可）のもとで使えそうなもの：

- **Canvas API**（ブラウザ組み込み）: ピクセル解析によるテキスト領域 vs 画像領域の粗い検出
- **OpenCV.js**: 輪郭検出・領域分割（ファイルサイズが重いのが課題）
- **Tesseract.js**: クライアントサイドOCR（Claudeのビジョンと役割分担できるか要検討）
- **Claude structured output**: Step 1 でクラスターの言語的記述を JSON 形式で返させる
- **他社 Vision API**（Google Vision 等）: バウンディングボックスを返せる → トリミング処理が可能になる
